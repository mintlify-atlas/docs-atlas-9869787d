The low-code CDK lets you define a REST API source connector as a YAML manifest. The CDK runtime interprets the manifest at sync time — no custom Python code required. This approach is well-suited for APIs that follow standard patterns: JSON responses, common authentication schemes, and page- or cursor-based pagination.

## Manifest structure

A low-code manifest is a YAML file with the following top-level keys:

```yaml
type: DeclarativeSource
version: "7.10.0"

check:
  type: CheckStream
  stream_names:
    - users

streams:
  - <stream definitions>

spec:
  type: Spec
  connection_specification:
    type: object
    required:
      - api_key
    properties:
      api_key:
        type: string
        title: API Key
        airbyte_secret: true
```

| Key | Description |
|---|---|
| `type` | Always `DeclarativeSource` for low-code connectors. |
| `version` | The CDK version this manifest targets (current: `7.10.0`). |
| `check` | Identifies which stream(s) to use when testing the connection. |
| `streams` | The list of stream definitions (see below). |
| `spec` | The connector's configuration schema rendered in the UI setup form. |

## Defining streams

Each stream describes how to fetch one endpoint from the API:

```yaml
streams:
  - type: DeclarativeStream
    name: users
    primary_key: id
    retriever:
      type: SimpleRetriever
      requester:
        type: HttpRequester
        url_base: "https://api.example.com"
        path: "/v2/users"
        http_method: GET
        authenticator:
          type: ApiKeyAuthenticator
          header: "Authorization"
          api_token: "Bearer {{ config['api_key'] }}"
      record_selector:
        type: RecordSelector
        extractor:
          type: DpathExtractor
          field_path:
            - data
            - items
      paginator:
        type: DefaultPaginator
        pagination_strategy:
          type: CursorPagination
          cursor_value: "{{ response['next_cursor'] }}"
          stop_condition: "{{ response['next_cursor'] is none }}"
        page_token_option:
          type: RequestOption
          inject_into: request_parameter
          field_name: cursor
    schema_loader:
      type: InlineSchemaLoader
      schema:
        $schema: "http://json-schema.org/draft-07/schema#"
        type: object
        properties:
          id:
            type: integer
          name:
            type: string
          email:
            type: string
```

## Requester

The `HttpRequester` component controls how the CDK builds HTTP requests:

```yaml
requester:
  type: HttpRequester
  url_base: "https://api.example.com"
  path: "/v2/{{ stream_slice['endpoint'] }}"
  http_method: GET
  request_headers:
    Accept: "application/json"
  request_parameters:
    limit: "100"
  authenticator:
    type: BearerAuthenticator
    api_token: "{{ config['access_token'] }}"
```

The path and parameters support Jinja2 templating. You can reference:

- `{{ config['field'] }}` — values from the user-supplied configuration
- `{{ stream_slice['key'] }}` — values from the current partition when using a partition router
- `{{ stream_interval['start_time'] }}` — incremental sync window boundaries
- `{{ response['field'] }}` — values extracted from the previous response (for cursor pagination)

## Authentication

The CDK supports several built-in authenticators:

  <Accordion title="API Key">
    ```yaml
    authenticator:
      type: ApiKeyAuthenticator
      header: "X-API-Key"
      api_token: "{{ config['api_key'] }}"
    ```
  </Accordion>

  <Accordion title="Bearer token">
    ```yaml
    authenticator:
      type: BearerAuthenticator
      api_token: "{{ config['access_token'] }}"
    ```
  </Accordion>

  <Accordion title="Basic HTTP">
    ```yaml
    authenticator:
      type: BasicHttpAuthenticator
      username: "{{ config['username'] }}"
      password: "{{ config['password'] }}"
    ```
  </Accordion>

  <Accordion title="OAuth 2.0">
    ```yaml
    authenticator:
      type: OAuthAuthenticator
      client_id: "{{ config['client_id'] }}"
      client_secret: "{{ config['client_secret'] }}"
      refresh_token: "{{ config['refresh_token'] }}"
      token_refresh_endpoint: "https://api.example.com/oauth/token"
    ```

## Pagination

The `DefaultPaginator` supports three pagination strategies:

<Tabs>
  <Tab title="Cursor pagination">
    Use when the API returns a token or URL pointing to the next page:

    ```yaml
    paginator:
      type: DefaultPaginator
      pagination_strategy:
        type: CursorPagination
        cursor_value: "{{ response['meta']['next_cursor'] }}"
        stop_condition: "{{ response['meta']['next_cursor'] is none }}"
      page_token_option:
        type: RequestOption
        inject_into: request_parameter
        field_name: cursor
    ```
  </Tab>

  <Tab title="Offset pagination">
    Use when the API uses an offset/limit pattern:

    ```yaml
    paginator:
      type: DefaultPaginator
      pagination_strategy:
        type: OffsetIncrement
        page_size: 100
      page_token_option:
        type: RequestOption
        inject_into: request_parameter
        field_name: offset
      page_size_option:
        type: RequestOption
        inject_into: request_parameter
        field_name: limit
    ```
  </Tab>

  <Tab title="Page increment">
    Use when the API uses a page number:

    ```yaml
    paginator:
      type: DefaultPaginator
      pagination_strategy:
        type: PageIncrement
        page_size: 50
        start_from_page: 1
      page_token_option:
        type: RequestOption
        inject_into: request_parameter
        field_name: page
    ```
  </Tab>
</Tabs>

## Record selection

The `DpathExtractor` navigates a JSON path to find the array of records in the response:

```yaml
record_selector:
  type: RecordSelector
  extractor:
    type: DpathExtractor
    field_path:
      - data      # response.data
      - items     # response.data.items
```

Leave `field_path` empty (`[]`) if the response body itself is the array of records.

## Incremental sync

To enable incremental syncs, add a `DatetimeBasedCursor` to your stream's retriever:

```yaml
incremental_sync:
  type: DatetimeBasedCursor
  cursor_field: updated_at
  datetime_format: "%Y-%m-%dT%H:%M:%SZ"
  start_datetime:
    type: MinMaxDatetime
    datetime: "{{ config['start_date'] }}"
    datetime_format: "%Y-%m-%d"
  end_datetime:
    type: MinMaxDatetime
    datetime: "{{ now_utc() }}"
    datetime_format: "%Y-%m-%dT%H:%M:%SZ"
  step: P30D
  cursor_granularity: PT1S
  start_time_option:
    type: RequestOption
    inject_into: request_parameter
    field_name: updated_since
```

## Testing

Run acceptance tests from the connector directory:

```bash
# Install test dependencies
pip install pytest connector-acceptance-test

# Run acceptance tests
python -m pytest integration_tests/ -v
```

You can also test locally using the `airbyte-cdk` CLI:

```bash
pip install airbyte-cdk==7.10.0

# Test spec
airbyte-cdk spec --config ./secrets/config.json

# Test check
airbyte-cdk check --config ./secrets/config.json

# Test discover
airbyte-cdk discover --config ./secrets/config.json

# Test read
airbyte-cdk read \
  --config ./secrets/config.json \
  --catalog ./integration_tests/configured_catalog.json
```

## Schema inference

If you do not want to write the stream schema by hand, the CDK can infer it from sample records. In the Connector Builder UI, click **Detect schema** after fetching test records. You can also use `InlineSchemaLoader` with `additionalProperties: true` as a permissive starting point and tighten it later:

```yaml
schema_loader:
  type: InlineSchemaLoader
  schema:
    $schema: "http://json-schema.org/draft-07/schema#"
    type: object
    properties: {}
    additionalProperties: true
```
