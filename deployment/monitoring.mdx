Airbyte emits metrics via [Micrometer](https://micrometer.io/), exposes HTTP health endpoints on every service, and uses Temporal for durable workflow execution that you can inspect through the Temporal UI.

## Enabling metrics

Metrics are disabled by default. Enable them in `values.yaml` and choose a backend.

### StatsD / Datadog

```yaml
global:
  metrics:
    enabled: true
    step: "PT1M"  # publish every minute

    statsd:
      enabled: true
      flavor: datadog  # or influxdb
      # host defaults to the node IP (status.hostIP) via the Kubernetes downward API
      port: 8125
```

Environment variables set by the chart:

| Variable | Value |
|---|---|
| `MICROMETER_METRICS_ENABLED` | `true` |
| `MICROMETER_METRICS_STATSD_ENABLED` | `true` |
| `MICROMETER_METRICS_STATSD_FLAVOR` | `datadog` |
| `STATSD_HOST` | node IP |
| `STATSD_PORT` | `8125` |

### OpenTelemetry (OTLP)

```yaml
global:
  metrics:
    enabled: true

    otlp:
      enabled: true
      collectorEndpoint: "http://otel-collector.monitoring:4317"

    otel:
      exporter:
        name: otlp
        protocol: grpc
        timeout: 30000          # ms
        metricExportInterval: 10000  # ms
```

Environment variables set by the chart:

| Variable | Value |
|---|---|
| `MICROMETER_METRICS_OTLP_ENABLED` | `true` |
| `OTEL_COLLECTOR_ENDPOINT` | your collector URL |

## Public metric names

These metrics are considered stable and are safe to use in dashboards and alerts:

| Metric | Type | Description |
|---|---|---|
| `api_requests` | counter | Total requests to the Airbyte API. |
| `gb_moved` | gauge | Gigabytes moved during a sync job. |
| `syncs_duration` | gauge | Sync duration in seconds. |
| `syncs` | counter | Count of syncs by status (tagged by `status`). |

## Metric tags

Metrics are tagged with attributes from `MetricTags`. Commonly used tags:

| Tag | Values | Description |
|---|---|---|
| `connection_id` | UUID | The connection involved in the operation. |
| `job_id` | integer | Job identifier. |
| `job_status` | `succeeded`, `failed`, `cancelled` | Terminal job status. |
| `attempt_outcome` | `succeeded`, `failed` | Attempt result. |
| `failure_origin` | `source`, `destination`, `airbyte_platform`, `unknown` | Where the failure occurred. |
| `failure_type` | `system_error`, `config_error` | Failure classification. |
| `workspace_id` | UUID | Workspace identifier. |
| `organization_id` | UUID | Organization identifier. |
| `data_plane_id` | UUID | Data plane identifier. |
| `release_stage` | `alpha`, `beta`, `generally_available` | Connector release stage. |

## Health check endpoints

All Java services expose a Micronaut management endpoint on port `8085`:

```
GET http://<service>:8085/health
```

A healthy response returns HTTP `200` with a JSON body:

```json
{ "status": "UP" }
```

The server additionally exposes its main API on port `8000`. Kubernetes liveness and readiness probes for the server are configured in `values.yaml` under `server.livenessProbe` and `server.readinessProbe`.

The webapp (nginx) responds on port `8080`.

## Temporal workflow monitoring

The bundled Temporal UI is available as the `airbyte-temporal-ui` deployment. To access it:

```bash
kubectl port-forward -n airbyte svc/airbyte-temporal-ui-svc 8088:8080
```

Then open `http://localhost:8088` to browse workflows, view execution history, and inspect failures.

For Temporal Cloud integrations, point `global.temporal.cloud.host` at your Temporal Cloud namespace endpoint and supply the client certificate and key:

```yaml
global:
  temporal:
    cloud:
      enabled: true
      host: "<namespace>.tmprl.cloud:7233"
      namespace: "<namespace>.<account>"
      clientCertSecretKey: TEMPORAL_CLOUD_CLIENT_CERT
      clientKeySecretKey: TEMPORAL_CLOUD_CLIENT_KEY
```

## Log aggregation

Connector logs are written to the configured object storage backend (`STORAGE_BUCKET_LOG`). You can access them through the Airbyte UI under a connection's **Job history** or via the API.

For structured log aggregation across all services, forward pod logs to your log aggregation stack using your cluster's standard tooling:

<Tabs>
  <Tab title="Fluent Bit">
    Deploy Fluent Bit as a DaemonSet and configure an output plugin pointing to Elasticsearch, Datadog, or any other destination. Airbyte services log to stdout/stderr in JSON format when `LOG_LEVEL=INFO`.
  </Tab>
  <Tab title="Datadog Agent">
    Enable the Datadog Agent DaemonSet and set `global.datadog.enabled: true` in `values.yaml`. The Datadog integration also collects StatsD metrics from the DogStatsD endpoint.

    ```yaml
    global:
      datadog:
        enabled: true
        env: production
        traceAgentPort: 8126
        statsd:
          port: 8125
    ```
  </Tab>
  <Tab title="OpenTelemetry">
    Deploy the OpenTelemetry Collector as a sidecar or DaemonSet. Configure the `otlp` metrics exporter in `values.yaml` (see above) and route logs via a Filelog receiver.
  </Tab>
</Tabs>

## Alerting recommendations

Set up alerts on these conditions to catch problems early:

| Alert | Condition | Severity |
|---|---|---|
| Jobs not completing | `num_pending_jobs` > threshold for > 30 min | Warning |
| Oldest pending job age | `oldest_pending_job_age_secs` > 3600 | Critical |
| Sync failure rate | `syncs{status=failed}` rate > 10% | Warning |
| Workload queue depth | `workload_queue_size` > 100 | Warning |
| Image pull failures | `workload_launcher_image_pull_failure` > 0 | Warning |
| Temporal workflow failures | `temporal_workflow_failure` rate > 0 | Critical |
